# Product

## Register

product

## Users

Gridix is used by real-estate developers, sales managers, agency partners, CRM users, and internal operators. They work inside project setup, inventory selection, lead handling, partner workflows, widget configuration, and document-heavy application flows.

Users need a calm, dense, predictable tool that lets them scan project state, act on leads, publish inventory, configure widgets, manage access, and move between admin and public project views without re-learning controls on each screen.

## Product Purpose

Gridix helps real-estate teams publish interactive project inventory, manage apartments or objects, connect CRM workflows, configure embeddable widgets, and support partner sales programs.

Design success means the interface feels trustworthy during repeated operational use: users can find the right object, understand state, complete setup flows, recover from errors, and confidently make destructive or publishing changes.

## Brand Personality

Calm, precise, capable.

The interface should feel like a serious operations product rather than a decorative real-estate landing page. It can be polished and premium, but visual choices must serve task clarity, scanning, and confidence.

## Anti-references

Avoid generic AI SaaS surfaces: over-rounded cards, dark-black sidebars with harsh white content, repeated bordered cards, blue/purple gradients, decorative glass, random icon styles, and modal-heavy flows.

Avoid browser-native confirmations, unstructured long forms, inconsistent buttons, hardcoded color patches, and screen-level spinners where layout-aware loading states would preserve context.

## Design Principles

1. Make state visible: loading, saving, empty, error, inactive, read-only, selected, and destructive states should be explicit.
2. Optimize for scanning: dense data, stable tables, compact toolbars, visible filters, and predictable action placement beat decorative layouts.
3. Keep one component language: the same action should look and behave the same across admin, editor, CRM, widgets, and partner flows.
4. Prefer progressive flows over interruptions: use inline sections, sheets, wizards, and confirmation dialogs with context before reaching for large modals.
5. Let the product feel premium through restraint: typography, spacing, color discipline, and motion quality should replace ornamental styling.

## Accessibility & Inclusion

Target WCAG 2.1 AA for product surfaces. Preserve keyboard navigation, visible focus states, dialog focus trapping, reduced-motion compatibility, readable contrast, mobile touch targets, and text that survives translation across supported locales.
