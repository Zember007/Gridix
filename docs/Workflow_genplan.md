# Логика Workflow с подпроектами и Генпланом

## Контекст

Ранее система работала как **смарт-каталог (шахматка)** для застройщиков, где:

- Был **один проект**
- Внутри проекта:
  - Редактирование фасада
  - Редактирование апартаментов
  - Настройка полей
  - Шахматка

То есть вся логика находилась **в одном уровне проекта**.

---

## Новая задача

Добавить **Генплан**, при этом:

- Не сломать текущий Workflow
- Не усложнить интерфейс для старых пользователей
- Поддержать несколько независимых структур внутри одного проекта

---

## Новая архитектура: Проект → Подпроекты

Теперь вводится новая сущность:

**Проект**
→ **Подпроекты**
→ Данные (фасады, квартиры, поля и т.д.)

---

## Основная логика

### 1. При создании проекта

Когда пользователь создает **проект**, автоматически создается:

- **Первый подпроект (дефолтный)**
- В него переносится весь текущий функционал

То есть:

**Проект**
└── Подпроект (Default / Основной)
  ├── Фасад
  ├── Апартаменты
  ├── Поля
  └── Шахматка

Для пользователя это выглядит **как старый Workflow**.

Ничего визуально не меняется.

---

## 2. Миграция старых проектов

Для существующих проектов:

- Создается подпроект автоматически
- Все данные переносятся внутрь него

До:

```
Проект
 ├── Фасад
 ├── Апартаменты
 └── Поля
```

После:

```
Проект
 └── Подпроект (Migrated)
     ├── Фасад
     ├── Апартаменты
     └── Поля
```

Для пользователей изменения **невидимы**.

---

## 3. При активации Генплана

Когда пользователь включает **Генплан**:

- Текущий "основной" проект становится **подпроектом**
- Появляется уровень управления подпроектами

Структура становится:

```
Проект
 ├── Подпроект 1 (Корпус A)
 ├── Подпроект 2 (Корпус B)
 └── Подпроект 3 (Башня)
```

Каждый подпроект имеет:

- Свой фасад
- Свои квартиры
- Свои поля
- Свою шахматку

---

## 4. Независимость подпроектов

Важно:

Каждый подпроект полностью независим.

Например:

Подпроект 1:

- 10 этажей
- 120 квартир
- Один фасад

Подпроект 2:

- 25 этажей
- 300 квартир
- Совершенно другой фасад

Данные **не пересекаются**.

---

## 5. UX-логика

До Генплана:

- Пользователь работает как обычно
- Не видит подпроектов

После Генплана:

- Появляется переключение подпроектов
- Каждый подпроект редактируется отдельно

---

## 6. Итоговый Workflow

### Создание проекта

```
Создать проект
↓
Создается подпроект (Default)
↓
Редактирование как раньше
```

---

### Включение Генплана

```
Активировать Генплан
↓
Текущий проект становится подпроектом
↓
Можно создавать новые подпроекты
↓
Каждый подпроект независим
```

---

## Почему это хорошее решение

✅ Не ломает старый Workflow
✅ Поддерживает Генплан
✅ Масштабируется
✅ Простая миграция
✅ Понятная логика
