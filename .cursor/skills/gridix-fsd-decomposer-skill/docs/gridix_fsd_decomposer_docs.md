# 🤖 Gridix FSD Decomposer

Инструмент для безопасной декомпозиции "толстых" React‑компонентов в FSD‑структуру.

Работает по принципу: **PLAN → REVIEW → APPLY EXACT**.

---

## 🎯 Зачем нужен

- Разделять UI, entity‑логику и feature‑логику
- Переносить legacy из `components/*` в `features/*` и `entities/*`
- Делать рефакторинг предсказуемым и контролируемым
- Генерировать готовый commit plan и PR draft

---

## ⚙️ Как работает

### 1️⃣ PLAN (по умолчанию)

```text
Use skill gridix-fsd-decomposer.
PLAN for apps/main/src/components/admin/LeadsManager.tsx
```

Skill:

- **сначала проверяет** — нужна ли декомпозиция вообще (NO-OP gate)
- если файл уже в правильном слое и хорошо структурирован — скажет "декомпозиция не нужна" без лишнего плана
- если нужен только перенос (файл чистый, но в `components/`) — выдаст лёгкий план (move + importers)
- для файлов >400 строк — запускает **субагенты** (параллельно: структурный скан + поиск импортёров)
- классифицирует части (Feature / Entity / Shared UI / Shared Lib / Types)
- оценивает сложность миграции (**S** / **M** / **L**)
- предлагает новую структуру
- выводит список файлов для изменения (**EXACT APPLY FILE LIST**)
- находит все файлы-импортёры через `grep`/`rg` (**Affected Importers**)
- генерирует commit plan и PR draft

❗ На этом этапе код НЕ меняется.

---

### 2️⃣ REVIEW

Проверь:

- корректно ли выбран слой
- не трогает ли `pages/*`
- не уходит ли в `packages/ui` (запрещено без `ALLOW_PACKAGES_UI_PROMOTION`)
- нет ли widgets (запрещено без `ALLOW_WIDGETS`)
- адекватный ли список файлов
- все ли импортёры учтены
- нет ли лишних изменений
- нет ли cross-entity импортов

Если нужно изменить план:

```text
ADJUST: ...
```

---

### 3️⃣ APPLY

```text
APPLY
```

Skill:

- изменяет ТОЛЬКО файлы из утверждённого списка
- не трогает ничего вне плана
- если нужно выйти за рамки — остановится и попросит подтверждение
- если при APPLY обнаружен новый импортёр — остановится и запросит delta-approval

---

## 📦 Куда что переносится

| Тип                                   | Куда                 |
| ------------------------------------- | -------------------- |
| use-case, orchestration, side-effects | `features/*`         |
| entity UI + read logic                | `entities/*`         |
| чистый UI                             | `shared/ui/*`        |
| утилиты / хелперы / константы         | `shared/lib/*`       |
| кросс-app типы                        | `packages/types`     |
| локальные доменные типы               | `entities/*/model/`  |
| page-композиция                       | остаётся в `pages/*` |

---

## 🔐 Escape-флаги

По умолчанию запрещено:

| Действие                | Флаг для разрешения           |
| ----------------------- | ----------------------------- |
| Создание `widgets/`     | `ALLOW_WIDGETS`               |
| Перенос в `packages/ui` | `ALLOW_PACKAGES_UI_PROMOTION` |

Добавьте флаг в задачу, если действие необходимо.

---

## 🤖 Стратегия для больших файлов (>400 строк)

Для жирных файлов скилл автоматически использует **субагенты**:

| Субагент        | Задача                                             | Тип       |
| --------------- | -------------------------------------------------- | --------- |
| Structure scan  | Экспорты, хуки, компоненты, импорты, секции файла  | `explore` |
| Importer search | Все файлы в репо, которые импортируют целевой файл | `explore` |

Оба запускаются **параллельно**. Главный агент получает компактные отчёты и на их основе строит план.

Для APPLY с >10 файлами — изменения применяются через субагенты пакетами (по логическим группам).

Для файлов ≤400 строк субагенты не нужны — работает напрямую.

---

## 🧠 Когда использовать

- Компонент >300 строк
- Смешаны UI + запросы + side-effects
- Legacy в `components/*`
- Много deep imports

---

## 🚫 Когда не использовать

- Маленький UI-компонент
- CSS-правка
- Срочный hotfix
- Изменение только одной строки

> **Но можно кидать любой файл** — скилл сам определит, нужна ли декомпозиция. Если файл уже правильно расположен и структурирован, скилл ответит "декомпозиция не нужна" и объяснит почему. Лишний план не сгенерируется.

## 📝 Пример

### Было

`apps/main/src/components/admin/LeadsManager.tsx`

Внутри:

- таблица лидов
- фильтры
- модалка назначения
- мутация assignLead
- toast + invalidateQueries

### PLAN предложит

- `features/assign-lead/` (мутация + orchestration + toast) — **M**
- `entities/lead/ui/LeadRow.tsx` (представление строки) — **S**
- `shared/ui/TableSection.tsx` (переиспользуемая таблица без доменной логики) — **S**
- thin-wrapper в `components/admin/LeadsManager.tsx` (временно, с `// TODO: remove wrapper`)

### После APPLY

- логика assign вынесена в `features/assign-lead`
- UI строки лида — в `entities/lead/ui`
- общая таблица — в `shared/ui`
- компонент в `components/` стал thin-wrapper или удалён

Поведение не изменилось, структура стала FSD‑корректной.

---

## ⚠️ Правила команды

- Нельзя делать APPLY без ревью плана
- Не запускать на большие папки без согласования
- Не трогать auth/RLS/billing
- Cross-entity импорты запрещены (entities не должны импортировать друг друга)
- Thin-wrapper в `components/` обязан иметь `// TODO: remove wrapper` — запланируйте удаление

---

## ✅ Definition of Done

После APPLY:

- lint / typecheck / build проходят
- Поведение не изменилось
- Нет нарушений FSD-направления (shared → entities → features → pages)
- Нет новых циклических импортов
- Нет cross-entity импортов
- Хуки корректны (deps, без условных вызовов)
- Тесты (если были) перенесены и обновлены
- Co-located стили перенесены вместе с компонентом

---

Этот инструмент — помощник, а не автопилот.
Всегда сначала читаем PLAN.

## 📌 Шаблон постановки задачи скилу

Используйте следующий формат при работе со скиллом:

```text
Используй skill gridix-fsd-decomposer.

PLAN для:
<путь_к_файлу_или_папке>

Ограничения:
- Не изменять pages/*
- Не изменять packages/*
- Сохранить текущее поведение
- Использовать только Supabase wrapper (без прямого клиента)
- Соблюдать формат PR из репозитория

Цель:
<что именно нужно сделать>
```

После получения плана:

1. Проверить список файлов (EXACT APPLY FILE LIST).
2. Проверить Affected Importers — все ли учтены.
3. Убедиться, что объём изменений адекватный.
4. При необходимости скорректировать план через `ADJUST`.
5. Написать `APPLY` только после проверки.
