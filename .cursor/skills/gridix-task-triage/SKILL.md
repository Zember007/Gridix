# Gridix Task Triage

Преобразует сырое описание задачи в короткий, одинаковый для команды SPEC.

## Output contract (обязателен)

- Язык ответа: только русский.
- Формат по умолчанию: compact (без длинной теории).
- Полный разбор: только по явному запросу пользователя.
- В каждом ответе сохраняй структуру:
  - `Goal`
  - `Context/Route`
  - `Scope`
  - `DoD`
  - `Repro` (если это баг)
  - `Questions` или `Assumptions`

## Apply gates (обязательны)

- Этот skill работает в режиме анализа и подготовки SPEC.
- Любое действие, ведущее к изменению файлов, требует явного сигнала пользователя `APPLY`.
- Если пользователь просит корректировки SPEC, используй gate `ADJUST` и выдай обновленный SPEC без изменения файлов.

## Вход

- Сырая задача (текст, тикет, комментарии) и/или скриншот.

## Алгоритм

1. Выдели цель задачи в 1-2 предложениях.
2. Определи предполагаемый маршрут (`app/route`, экран, фича, слой).
3. Ограничь scope: что входит и что явно не входит.
4. Сформируй DoD как проверяемые критерии.
5. Для бага: добавь шаги воспроизведения (`Repro`).
6. Если данных не хватает:
   - задай не более 3 вопросов;
   - если нужно больше, зафиксируй рабочие `Assumptions` и продолжай.

## Формат ответа (compact)

```md
Goal: ...
Context/Route: ...
Scope:

- in: ...
- out: ...
  DoD:
- ...
  Repro:
- ...
  Questions:
- ...
```

Если вопросов нет, замени блок `Questions` на `Assumptions`.

## Stop conditions

- Противоречивые требования.
- Неясно, это bugfix или новая функциональность.
- Требуется доступ к данным/среде, которых нет в контексте.

В этих случаях зафиксируй 1-3 уточнения и остановись до ответа пользователя.
